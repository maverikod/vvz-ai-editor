# Исходное ТЗ — тонкий AI Editor Server

## Назначение

{8k2m} Проект **AI Editor Server** (`ai-editor-server`) — отдельный MCP-сервер, единственная задача которого: дать модели безопасный и предсказуемый цикл **редактирования файлов проекта** с опорой на **Code Analysis Server** (далее **СА**). Редактор не хранит каталог проектов, не ведёт собственную базу данных проектов и файлов, не индексирует код и не выполняет поиск.

{3p9n} Редактор **не управляет** жизненным циклом сессий СА. Сессию создаёт, держит и закрывает **модель (агент)** через API СА. Редактор **использует** уже существующую валидную сессию СА как обязательный контекст для любой работы с файлами.

{7w1c} Блокировки и учёт «кто держит файл» — зона **СА**. Редактор меняет блокировки и содержимое на диске проекта **только** через внутренний upstream-клиент (JSON-RPC / MCP Proxy к СА), не через свой MCP и не через прямой доступ к БД или дереву проекта СА. Команды `session_create` / `session_delete` вызывает **агент на СА**, не редактор.

## Роли систем

{4h8e} **Code Analysis Server** владеет: реестром проектов и файлов, **клиентскими сессиями** (`session_id`), блокировками файлов в БД, отдачей и приёмом содержимого файлов (chunked transfer), **блокировкой и разблокировкой** файла для сессии. На СА команды `session_open_file` и `session_close_file` означают **lock / unlock** записи в `session_file_locks`, а не «открытие файла для редактирования». Валидация `session_id` — через touch/existence-check на любой команде СА с обязательным `session_id` (ошибка `SESSION_NOT_FOUND` при отсутствии).

{2f5a} **AI Editor Server** владеет: каталогом `{workspace_root}/{session_id}/` на диске редактора, локальным редактированием (CST/draft), и **единственным** публичным workflow MCP: `universal_file_open`, `universal_file_edit`, `universal_file_write`, `universal_file_close` (+ preview). **Open / edit / write / close для модели — только команды редактора.** Редактор **не** выставляет отдельный «ид сессии редактирования» (group UUID): ключ сессии везде — `session_id` СА.

{9d3b} **Модель-агент** владеет: выбором проекта и файлов, созданием и удалением сессии на СА, последовательностью вызовов команд редактора и СА, решением о commit/save/close.

## Сессия СА и каталог редактора

{1c7k} Для начала работы агент получает на СА `session_id` (команды СА вне scope данного ТЗ, кроме требования их существования). Этот идентификатор передаётся во **все** команды редактора, работающие в контексте сессии.

{5e2j} Под сессию СА на диске редактора создаётся **один каталог** `{workspace_root}/{session_id}/`. Имя каталога **равно** `session_id` СА. Пока ни один файл не открыт, каталог может быть пустым (аналогия: редактор запущен, документы ещё не открыты).

{6a4g} **Все** действия редактора в рамках сессии СА — только внутри `{workspace_root}/{session_id}/` и его **подкаталогов**. Запрещено создавать рабочие файлы, исходники или каталоги редактирования **вне** этого дерева (в том числе рядом с файлом проекта на диске СА или в корне `workspace_root`).

{0d4f} Для каждого открытого файла — **поддерево** в `{session_id}/`: (а) **исходник**; (б) **подкаталог редактирования** с draft, lockfile, sidecar (`.cst_sidecar`), tree-temp артефактами — **все** пути внутри workspace ({f7i5}), не рядом с файлом проекта на СА.

{f7i5} Артефакты Internal Edit Engine (sidecar, tree-temp roots, `.write` lockfile) размещаются относительно Edit Subdirectory в File Subtree, не относительно канонического пути проекта на диске СА.

{8b1h} Одна сессия СА может одновременно удерживать блокировки и открытые файлы **нескольких** путей. У каждого файла — своё поддерево внутри общего каталога `{session_id}/`.

## Жизненный цикл файла (обязательный workflow)

{2k9p} Workflow файла — четыре стадии. На этапе миграции они реализуются командами `universal_file_open`, `universal_file_edit`, `universal_file_write`, `universal_file_close` (семантика ниже; переименование — в нижних уровнях плана).

**Стадия 1 — Open (`universal_file_open`).** {7m3q} Команда **редактора** (не СА). Внутри: (1) **Session Guard** — проверка `session_id` на СА; (2) для существующего файла: `session_open_file` (lock) + download; (3) для `create=true` ({c4f2}): upload содержимого на СА, затем lock + локальная копия; (4) каталог `{workspace_root}/{session_id}/`, исходник и подкаталог редактирования — только на диске редактора.

{c4f2} Параметр `create=true` в `universal_file_open`: файл ещё не существует в проекте на СА. Порядок: touch сессии → `project_file_transfer_upload_save` (создание/запись на СА) → `list_project_files` (получить `file_id`) → `session_open_file` (lock) → локальный Origin Snapshot и Edit Subdirectory в workspace. Без записи на диск проекта СА с хоста редактора.

**Стадия 2 — Edit (`universal_file_edit` и preview).** {9p2s} Правки применяются **только** к содержимому в подкаталоге редактирования. Исходник в каталоге сессии на этой стадии не меняется.

**Стадия 3 — Write (`universal_file_write`).** {3t6u} Перед RPC к СА — сравнение экспорта Edit Subdirectory с Origin Snapshot ({e6h4}). Равенство → no-op без upload. Иначе → upload на СА; при успехе — обновление Origin Snapshot. При ошибке СА локальное состояние не меняется.

{e6h4} Опциональный `write_mode=preview` возвращает **локальный** diff (sidecar/tree-temp/text) без upload на СА; `write_mode=commit` (или omit после preview) выполняет compare-and-upload. In-session lockfile `.write` — только для локального preview/diff, не gate перед RPC к СА.

**Стадия 4 — Close (`universal_file_close`).** {5v0w} Команда **редактора**. Внутри: (1) `session_close_file` на СА — **разблокировка** `(session_id, project_id, file_id)`; (2) при успехе — удаление поддерева файла на диске редактора; (3) при отсутствии открытых файлов — удаление `{workspace_root}/{session_id}/`.

## Валидация сессии и «битая» сессия

{1x5y} Единая точка проверки сессии — **Session Guard** ({d5g3}): перед open/edit/preview вызывает touch на СА (`session_list_file_locks`). `session_list` без id в продакшене не используется. Доступ — только при валидном `session_id`; иначе отказ (`SESSION_NOT_FOUND`, конфликт блокировки).

{d5g3} Session Guard — внутренняя сущность редактора (не MCP): классифицирует операции normal (open/edit/preview) vs terminating (write/close); вызывает upstream validate; при NOT_FOUND на terminating write — попытка upload, ответ с ошибкой СА, draft сохраняется; на close — best-effort unlock и локальная очистка.

{6z8a} Если сессия на СА отсутствует или признана невалидной, редактор **отказывает** во всех операциях, кроме: (а) **write** — попытка сохранить локальные изменения на СА, если контракт СА ещё допускает; (б) **close** — освобождение локальных ресурсов и best-effort unlock на СА. Любые edit/preview/open нового файла в такой сессии запрещены.

{4c2d} Редактор **сбрасывает** локальное состояние сессии (каталог `{session_id}` и связанные метаданные), если СА подтвердил отсутствие сессии, чтобы не оставлять «зомби»-каталоги.

## MCP-поверхность редактора

{8g7f} На MCP: `universal_file_open`, `universal_file_edit`, `universal_file_write`, `universal_file_close`, `universal_file_preview`, плюс **`health`** (единственная инфраструктурная команда). `queue_health` **не** регистрируется — у thin editor нет фоновых очередей. Дублирующие семейства запрещены.

{2j4h} Команды СА (`session_open_file`, `session_close_file`, `project_file_transfer_*`, `session_create`/`session_delete` для агента на СА) **не** регистрируются на MCP редактора. Редактор вызывает их **только внутри** `universal_file_*` через upstream-клиент. Локальные `session_*`, `subordinate_session_*`, дублирующий transfer/lock MCP на редакторе **удаляются**.

{a1c4} **Контракт API СА** (только upstream редактора): `session_create`/`session_delete` (агент); touch — `session_list_file_locks`; lock/unlock — `session_open_file`/`session_close_file`; transfer — `project_file_transfer_download_begin`, `upload_save`; метаданные — `list_project_files`, `list_projects` (только резолв id, не MCP редактора). **Read-only download без lock** — отдельный вызов transfer для one-shot preview ({5l1i}). `session_view`, `subordinate_session_*` не входят. `get_project_root` на редакторе **не используется** для file workflow.

{5l1i} **Preview** — только `universal_file_preview` редактора: read-only по локальной рабочей копии после open; опционально одноразовое чтение с СА без lock (отдельная ветка внутри preview, не отдельная MCP-команда СА).

## Что удаляется из текущей кодовой базы редактора

{9q3k} Из репозитория AI Editor убирается всё вне workflow: локальная БД/`client_sessions` (in-memory эмуляция СА), MCP `session_*` и `subordinate_session_*` на редакторе, `session_view`, дублирующие `project_file_transfer_*` на MCP редактора, editor group UUID в `universal_file_*`, воркеры, поиск/анализ как MCP. `list_projects` / `list_project_files` — **только** внутри upstream к СА, не как MCP редактора.

{7r6m} Допустим остаточный **внутренний** код (например CST для Python), если он вызывается только из единого edit-path и не регистрируется отдельными MCP-командами.

## Upstream к Code Analysis Server

{3s9n} Редактор подключается к СА через MCP Proxy (`code-analysis-server` или настроенный `server_id`) по JSON-RPC. Конфигурация: хост, порт, TLS, таймауты — в конфиге редактора (`code_analysis_server` / `upstream` секция); секреты не в git.

{1u2o} Любое действие, которое на СА меняет блокировку, содержимое файла на диске проекта или состояние сессии, выполняется **только** вызовом команды СА, а не прямым SQL или файловым доступом к дереву проекта СА.

## Ограничения и границы

{6w5p} Редактор **не** заменяет СА для: создания проектов, полнотекстового/семантического поиска, CST-дерева на стороне СА (`cst_*` команды СА), комплексного анализа, очередей фоновых задач СА.

{8y7q} Упаковка в Docker/Debian (`ai-editor-docker`) сохраняется; персистентным на хосте является только `workspace_root`, логи и конфигурация. Состояние сессий и блокировок переживает рестарт **только** на СА.

{4z1r} Клиентская библиотека `ai-editor-client` отражает тот же единый workflow, что и MCP редактора; дублирующие фасады (`EditorFileClient` vs `EditorFileMCPClient` vs `FileSessionClient` как равноправные API) должны быть сведены к одному контракту.

## Критерии приёмки (уровень продукта)

{2a8s} После реализации плана: (1) все файлы сессии лежат только под `{workspace_root}/{session_id}/`; (2) open загружает с СА в каталог сессии и создаёт подкаталог редактирования; (3) write без изменений — no-op без вызова СА; write с изменениями — upload и обновление исходника при успехе; (4) close снимает блокировку на СА и удаляет поддерево файла; (5) вторая сессия СА не может открыть тот же файл, пока первая держит блокировку; (6) в `help` нет дублирующих семейств команд.
